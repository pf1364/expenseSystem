# 问题修改记录

## 2026-06-10 每日补助与金额可信边界修复

### 1. 问题背景

前端虽然限制每日餐补、交通补助和通讯补助不能超过标准金额，但客户端请求默认不可信。用户可以绕过页面，使用 Postman 等工具直接修改请求 JSON。

检查发现，后端已经能够根据城市标准表校验单项补助金额，例如北京餐补标准为 100 元时，传入 `mealAmount=100000` 会被拒绝。但每日补助和分摊仍存在一些可构造数据的问题。

### 2. 发现的问题

#### 2.1 每日补助日期可以超出行程范围

原实现会保存客户端传入的 `allowanceDate`，但没有检查日期是否位于行程的开始日期和结束日期之间。

例如行程为 2026-04-13 至 2026-04-17，客户端可以额外提交 2026-04-18 的补助，从而抬高总金额。

#### 2.2 同一行程可以提交重复的补助日期

建表 SQL 已有 `(itinerary_id, allowance_date)` 唯一索引，但业务层没有提前校验。重复日期可能在金额预估阶段被重复累计，最终到数据库写入时才失败，错误信息也不友好。

#### 2.3 每日补助城市可以与行程目的地不一致

原实现直接使用每日明细中的 `cityCode/cityName` 查询补助标准。攻击者可以把低标准目的地替换成高标准城市。

#### 2.4 分摊比例和金额缺少单行边界校验

原实现主要校验所有分摊比例和金额的合计，没有阻止单行负金额、负比例或超过 100% 的比例。

#### 2.5 关键金额逻辑缺少说明

服务实现中虽然存在金额重算与上限校验，但缺少注释，不容易判断哪些字段信任前端、哪些字段以后端计算为准。

### 3. 修改内容

#### 3.1 新增每日补助与行程一致性校验

新增 `validateAllowanceDaysForItinerary`：

- 每日补助日期不能为空。
- 补助日期必须位于行程日期范围内，包含开始日和结束日。
- 同一行程内的补助日期必须唯一。
- 补助城市必须与行程目的地一致。
- 校验通过后，用城市标准表中的权威城市编码、名称和等级覆盖请求值。

该校验同时接入：

- 创建草稿
- 创建并提交
- 更新报销单
- 提交已有草稿

因此不能通过保存草稿绕过规则。

#### 3.2 保留并确认后端金额校验

每日补助保存时：

- 根据数据库城市标准重新取得餐补、交通补助和通讯补助上限。
- 未勾选的项目强制按 0 元计算。
- 已勾选项目金额必须大于等于 0。
- 已勾选项目金额不得超过数据库标准。
- `dayAmount` 由后端重新计算。
- 主表各项金额和总金额从已保存每日明细重新汇总。

前端传入的标准金额、当日合计和主表总金额均不作为可信计算结果。

#### 3.3 增加分摊单行边界校验

- 单条分摊比例必须位于 0 到 1，即页面展示的 0% 到 100%。
- 分摊金额不能小于 0。
- 正式提交时继续校验比例合计为 100%，金额合计等于报销总金额。

#### 3.4 增加代码注释

在以下关键位置增加中文注释：

- 子表差异同步
- 主表金额重新汇总
- 提交前校验
- 每日补助与行程一致性校验
- 分摊金额边界校验
- 后端总金额估算
- 城市标准及单项补助金额校验

### 4. 测试内容

新增后端测试，覆盖：

1. 补助日期超出行程范围时拒绝。
2. 同一行程存在重复补助日期时拒绝。
3. 补助城市与行程目的地不一致时拒绝。
4. 北京餐补标准为 100 元时，提交 100000 元被拒绝。

### 5. 本次暂不处理

按照当前业务约定，本次不校验不同或同一人员的多条行程是否重叠。行程重叠暂交由后续审核流程识别。

后续如果需要自动校验，可按“出行人 + 日期区间”查询已有有效行程，并结合单据状态、跨城中转和审批规则进行判断。

# 差旅报销单数据库设计说明

本文描述差旅报销单项目当前使用的业务数据库表。当前设计围绕报销单主数据、行程、每日补助、城市补助标准、费用分摊 5 张表展开。

查询一个报销单详情时，通常需要读取：

- `fk_reim_main`：报销单主表
- `fk_reim_itinerary`：行程表
- `fk_reim_allowance_day`：每日补助明细表
- `fk_reim_allocation`：费用归属及分摊表

`fk_city_allowance` 主要在新增/编辑行程时生成每日补助明细，以及保存补助日历时校验金额上限。

## 1、fk_reim_main 报销单主表

报销单主表用于保存一张差旅报销单的核心信息。列表页展示的大部分字段来自该表，详情页中的基础信息、费用合计、备注信息也来自该表。

一张报销单对应一条主表记录；一张报销单可以关联多条行程、多条每日补助明细和多条费用分摊记录。

## 2、fk_reim_itinerary 行程表

行程表用于保存报销单中的补录行程信息。一个报销单可以有多个行程，例如同一张报销单中可以有“武汉-北京”和“北京-上海”两段行程。

新增一条行程后，后端需要根据出发日期、到达日期、到达城市生成对应的每日补助明细。

## 3、fk_reim_allowance_day 每日补助明细表

每日补助明细表用于保存每天、每类补助的选择状态和金额。这个表是补助日历弹窗的核心数据来源。

例如一个行程是 2026-04-13 到 2026-04-17，共 5 天，那么会生成 5 条每日补助明细。用户可以只勾选某一天的餐补，也可以只勾选某一天的餐补和交通补助，所以必须按天保存明细。

## 4、fk_city_allowance 城市补助标准表

城市补助标准表用于保存城市等级和各类补助标准。后端在生成每日补助明细时，会根据到达城市查询该表，带出餐补、交通补助、通讯补助标准。

该表也是保存补助日历时的金额校验依据：用户输入的补助金额不能超过标准金额。

| meal_standard | decimal | 12,2 | Y |  | 餐费补助标准金额 | 文档规则：一线 100，二线 80，三线 50 |

## 5、fk_reim_allocation 费用归属及分摊表

费用归属及分摊表用于保存报销单的费用分摊信息。一个报销单至少保留一条分摊信息，也可以添加多条分摊。

费用归属可能是多个公司共同承担，也可能是多个部门共同承担，因此不能放在报销单主表里。主表只保存报销单自身信息，具体由哪些归属方承担、承担比例和金额，都保存在本表。

分摊比例在数据库中按 0 到 1 存储，页面展示时转换为百分比。例如数据库存 `0.500000`，页面展示为 `50.00%`。

| allocation_owner_type | varchar | 32 | Y | DEPARTMENT | 分摊归属方类型 | COMPANY 公司，DEPARTMENT 部门 |
| allocation_owner_id | varchar | 64 | Y |  | 分摊归属方ID | 前端写死费用归属选项的ID |
| allocation_owner_no | varchar | 64 | N | NULL | 分摊归属方编号 | 公司编号或部门编号 |
| allocation_owner_name | varchar | 100 | Y |  | 分摊归属方名称 | 公司名称或部门名称，保存快照 |
| business_id | varchar | 64 | N | NULL | 分摊业务ID | 前端写死业务选项的ID |
| business_name | varchar | 100 | N | NULL | 分摊业务名称 | 业务名称，保存快照 |
| allocation_ratio | decimal | 10,6 | Y | 0.000000 | 分摊比例 | 数据库存 0 到 1，页面展示为百分比 |
| allocation_amount | decimal | 12,2 | Y | 0.00 | 分摊金额 | 多条分摊金额合计应等于报销单补助总金额 |
| sort_no | int |  | Y | 1 | 排序号 | 控制页面分摊行展示顺序 |
| created_at | datetime |  | Y | CURRENT_TIMESTAMP | 创建时间 | 记录创建时间 |
| updated_at | datetime |  | Y | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 | 记录最后更新时间 |

## 补充说明

- 金额和比例字段在 Java 中统一使用 `BigDecimal`，避免 `double` 浮点误差影响金额计算和相等判断。
- 日期字段使用 `LocalDate`，时间字段使用 `LocalDateTime`。
- 训练营项目中人员、部门、公司、业务类型、项目等选项由前端写死，因此这里没有单独设计员工表、部门表、公司表、业务类型表、项目表。
- 虽然部分选项由前端写死，报销单主表和分摊表仍保存名称快照，避免后续前端选项调整后影响历史单据展示。

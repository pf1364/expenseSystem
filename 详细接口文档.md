# 差旅报销单详细接口文档

> 文档版本：2026-06-10  
> 后端服务：Spring Boot 3.2.10  
> 默认服务地址：`http://localhost:8081`  
> 前端 Axios 基础路径：`/api`

## 1. 文档说明

本文档严格根据当前项目中的 Controller、DTO、Service 实现和前端 API 调用编写，描述的是当前已经存在的 12 个接口。

### 1.1 统一响应格式

有返回数据时：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

没有返回数据时，由于 Jackson 配置了忽略 `null` 字段，实际响应通常为：

```json
{
  "code": 200,
  "message": "success"
}
```

业务参数错误：

```json
{
  "code": 400,
  "message": "报销标题不能为空"
}
```

未处理的系统异常：

```json
{
  "code": 500,
  "message": "error"
}
```

| 字段 | 类型 | 是否必有 | 说明 |
| --- | --- | --- | --- |
| code | Integer | 是 | 业务状态码：`200` 成功、`400` 参数或业务校验失败、`500` 系统异常 |
| message | String | 是 | 响应说明。成功固定为 `success` |
| data | Object/Array | 否 | 响应数据。值为 `null` 时通常不输出 |

**当前实现注意事项：**

- 全局异常处理器返回的是业务 `code`，没有通过 `ResponseEntity` 设置 HTTP 状态。
- 因此不少业务失败响应的 HTTP 状态仍可能是 `200 OK`，调用方必须同时判断响应体中的 `code`。
- 前端响应拦截器以 `code === 200` 判断成功，否则弹出 `message` 并拒绝 Promise。

### 1.2 通用格式

| 类型 | 格式 | 示例 |
| --- | --- | --- |
| 日期 | `yyyy-MM-dd` | `2026-04-13` |
| 日期时间 | ISO-8601 | `2026-06-10T14:30:25` |
| 金额 | JSON Number，数据库使用 `decimal`，业务保留两位小数 | `360.00` |
| 分摊比例 | 推荐传 `0-1` | `0.300000` 表示 30% |
| 选中标志 | Integer | `1` 选中，`0` 未选中 |

后端兼容大于 `1` 的百分数写法，例如传入 `30` 会被转换为 `0.300000`。新代码仍应统一传 `0-1`，避免歧义。

### 1.3 单据状态

| 编码 | 名称 | 说明 |
| --- | --- | --- |
| DRAFT | 草稿 | 允许删除；当前更新接口允许修改 |
| SUBMITTED | 已提交 | 已完成提交；当前代码仍允许通过更新接口修改 |
| VOIDED | 已作废 | 不允许修改 |

### 1.4 公共请求结构

#### 1.4.1 报销单保存请求 `ReimbursementSaveRequest`

| 字段 | 类型 | 草稿必填 | 提交必填 | 说明 |
| --- | --- | --- | --- | --- |
| reimNo | String | 否 | 否 | 报销单号。创建时后端忽略并重新生成；更新时以路径参数为准 |
| title | String | 否 | 是 | 报销标题 |
| reimburserId | String | 否 | 否 | 报销人 ID，当前固定选项场景可为空 |
| reimburserNo | String | 否 | 否 | 报销人工号 |
| reimburserName | String | 否 | 是 | 报销人姓名 |
| reimDepartmentId | String | 否 | 否 | 报销部门 ID |
| reimDepartmentNo | String | 否 | 否 | 报销部门编号 |
| reimDepartmentName | String | 否 | 是 | 报销部门名称 |
| reimCompanyNames | String | 否 | 否 | 费用归属公司名称汇总。保存后会根据公司分摊记录重新汇总 |
| businessTypeId | String | 否 | 否 | 业务类型 ID |
| businessTypeNo | String | 否 | 否 | 业务类型编号 |
| businessTypeName | String | 否 | 是 | 业务类型名称 |
| reason | String | 否 | 是 | 出差事由 |
| remark | String | 否 | 否 | 备注。当前前端页面已不展示，但后端仍接收 |
| itineraries | Array | 否 | 是 | 行程列表。提交时至少需要一条 |
| allocations | Array | 否 | 是 | 费用归属及分摊列表 |

> 当前提交校验没有单独校验 `reimCompanyNames`，但要求至少存在一条有效分摊数据。

#### 1.4.2 行程请求 `ItineraryRequest`

| 字段 | 类型 | 提交必填 | 说明 |
| --- | --- | --- | --- |
| id | Long | 否 | 行程主键。更新已有行程时传入；新增时不传 |
| travelerId | String | 否 | 出行人员 ID |
| travelerNo | String | 否 | 出行人员工号 |
| travelerName | String | 否 | 出行人员姓名 |
| startCityCode | String | 否 | 出发城市编码 |
| startCityName | String | 是 | 出发城市名称 |
| endCityCode | String | 否 | 到达城市编码 |
| endCityName | String | 是 | 到达城市名称，也是补助标准查询城市 |
| startDate | LocalDate | 是 | 出发日期 |
| endDate | LocalDate | 是 | 到达日期，不能早于出发日期 |
| days | Integer | 否 | 行程天数。后端根据日期重新计算，不信任前端值 |
| routeText | String | 否 | 行程展示文本。为空时后端生成“出发城市-到达城市” |
| description | String | 否 | 行程说明 |
| allowanceDays | Array | 否 | 每日补助明细。为空时后端根据日期范围和到达城市自动生成 |

#### 1.4.3 每日补助请求 `AllowanceDayRequest`

| 字段 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| id | Long | 否 | 明细主键。更新已有明细时传入 |
| allowanceDate | LocalDate | 是 | 补助日期，必须位于对应行程日期范围内；同一行程内不得重复 |
| weekName | String | 否 | 星期名称。为空时后端根据日期计算 |
| cityCode | String | 条件必填 | 补助城市编码，与城市名称至少应提供一个，并且必须对应行程目的地 |
| cityName | String | 条件必填 | 补助城市名称，与城市编码至少应提供一个，并且必须对应行程目的地 |
| cityLevel | Integer | 否 | 城市等级。后端根据城市标准表重新写入 |
| mealStandard | BigDecimal | 否 | 餐费补助标准。后端重新查询，不信任前端值 |
| mealSelected | Integer | 否 | 是否申请餐补：`1` 是、`0` 否。空值按 `0` |
| mealAmount | BigDecimal | 否 | 餐补实报金额，必须在 `0` 到城市标准之间 |
| trafficStandard | BigDecimal | 否 | 交通补助标准。后端重新查询 |
| trafficSelected | Integer | 否 | 是否申请交通补助 |
| trafficAmount | BigDecimal | 否 | 交通补助实报金额，不能超过标准 |
| communicationStandard | BigDecimal | 否 | 通讯补助标准。后端重新查询 |
| communicationSelected | Integer | 否 | 是否申请通讯补助 |
| communicationAmount | BigDecimal | 否 | 通讯补助实报金额，不能超过标准 |
| dayAmount | BigDecimal | 否 | 当日合计。后端重新计算，不信任前端值 |

未选中的补助项目会被强制保存为 `0.00`。选中但金额为空时，也按 `0.00` 处理。校验通过后，后端使用行程目的地在城市标准表中查询到的权威城市信息覆盖请求值。

#### 1.4.4 分摊请求 `AllocationRequest`

| 字段 | 类型 | 提交必填 | 说明 |
| --- | --- | --- | --- |
| id | Long | 否 | 分摊主键。更新已有分摊时传入 |
| allocationOwnerType | String | 否 | 归属类型，当前前端固定传 `COMPANY`；为空时后端默认 `COMPANY` |
| allocationOwnerId | String | 否 | 归属公司 ID |
| allocationOwnerNo | String | 否 | 归属公司编号 |
| allocationOwnerName | String | 是 | 归属公司名称。为空的草稿分摊行会被跳过 |
| businessId | String | 否 | 分摊项目或业务 ID |
| businessName | String | 否 | 分摊项目或业务名称 |
| allocationRatio | BigDecimal | 是 | 分摊比例，推荐范围 `0-1`，合计必须为 `1` |
| allocationAmount | BigDecimal | 是 | 分摊金额，合计必须等于报销总金额，允许 `0.01` 误差 |
| sortNo | Integer | 否 | 排序号。为空时后端按请求顺序从 1 生成 |

## 2. 报销单分页查询

### 2.1 基本信息

- 功能：按组合条件分页查询报销单，按创建时间倒序返回。
- 方法：`GET`
- 路径：`/api/reimbursements/page`

### 2.2 Query 参数

| 参数 | 类型 | 必填 | 查询方式 | 说明 |
| --- | --- | --- | --- | --- |
| pageNum | Integer | 否 | - | 当前页，默认 1 |
| pageNo | Integer | 否 | - | 当前页别名；同时传入时优先于 `pageNum` |
| pageSize | Integer | 否 | - | 每页条数，默认 10；当前没有最大值限制 |
| reimNo | String | 否 | 模糊 | 报销单号 |
| title | String | 否 | 模糊 | 报销标题 |
| reason | String | 否 | 模糊 | 出差事由 |
| reimCompanyName | String | 否 | 模糊 | 主表冗余字段中的费用归属公司名称 |
| reimDepartmentName | String | 否 | 精确 | 报销部门名称 |
| reimburserKeyword | String | 否 | 精确 OR | 同时匹配报销人姓名或工号 |
| reimburserName | String | 否 | 精确 | 报销人姓名 |
| reimburserNo | String | 否 | 精确 | 报销人工号 |
| businessTypeName | String | 否 | 精确 | 业务类型名称 |
| billStatus | String | 否 | 精确 | `DRAFT`、`SUBMITTED` 或 `VOIDED` |

请求示例：

```http
GET /api/reimbursements/page?pageNo=1&pageSize=10&reimNo=100&reimburserKeyword=徐年年&billStatus=DRAFT
```

### 2.3 响应字段

`data` 为分页对象：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| total | Long | 符合条件的总记录数 |
| pageNum | Integer | 当前页 |
| pageSize | Integer | 每页条数 |
| records | Array | 当前页报销单列表 |

`records` 元素：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | Long | 主表 ID |
| reimNo | String | 报销单号 |
| billStatus | String | 状态编码 |
| billStatusName | String | 状态名称 |
| billTypeName | String | 单据类型名称 |
| reimburserName | String | 报销人姓名 |
| reimburserNo | String | 报销人工号 |
| reimDepartmentName | String | 报销部门名称 |
| reimCompanyNames | String | 费用归属公司名称汇总 |
| businessTypeName | String | 业务类型名称 |
| title | String | 报销标题 |

## 4. 创建草稿

### 4.1 基本信息

- 功能：创建一张草稿报销单，后端生成唯一单号。
- 方法：`POST`
- 路径：`/api/reimbursements/draft`
- Body：1.4.1 所述保存请求。

草稿允许基础字段不完整。空行程或缺少城市、日期的行程会被跳过；缺少归属公司名称的分摊行也会被跳过。

请求示例：

```json
{
  "title": "北京出差",
  "reimburserName": "徐年年",
  "reimDepartmentName": "客户成功事业部",
  "businessTypeName": "客户现场支持",
  "reason": "客户支持",
  "itineraries": [],
  "allocations": []
}
```

### 4.2 响应字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| reimNo | String | 后端生成的新报销单号 |
| billStatus | String | 固定为 `DRAFT` |
| billStatusName | String | 固定为“草稿” |

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "reimNo": "CLBX202606100001",
    "billStatus": "DRAFT",
    "billStatusName": "草稿"
  }
}
```

### 4.3 可能的错误

草稿本身不校验必填字段，但保存有效行程或补助时仍可能报错：

- `行程开始日期和结束日期不能为空`
- `行程结束日期不能早于开始日期`
- `每日补助日期不能为空`
- `每日补助日期必须在行程日期范围内：{日期}`

## 5. 创建并提交

### 5.1 基本信息

- 功能：一次性创建完整报销单并将状态设置为已提交。
- 方法：`POST`
- 路径：`/api/reimbursements/submit`
- Body：1.4.5 完整保存请求。

### 5.2 提交校验

- 标题、报销人姓名、报销部门名称、业务类型名称、出差事由必填。
- 至少一条行程，行程城市和日期完整，结束日期不早于开始日期。
- 每日补助为空时自动生成；已有每日补助时逐项校验金额。
- 分摊不能为空，归属名称不能为空。
- 分摊比例合计必须为 `1`，误差不超过 `0.000001`。
- 分摊金额合计必须等于补助总额，误差不超过 `0.01`。

### 5.3 响应示例

```json
{

  "data": {

    "billStatus": "SUBMITTED",
    "billStatusName": "已提交"
  }
}
```

提交成功且 Kafka 开关开启时，发送主题 `reim.bill.submitted`，消息内容为报销单号。

## 6. 更新报销单

### 6.1 基本信息

- 功能：更新主表、行程、每日补助和费用分摊。
- 方法：`PUT`
- 路径：`/api/reimbursements/{reimNo}`
- Body：1.4.1 所述保存请求。

路径中的 `reimNo` 为准，Body 中的同名字段会被覆盖。

子表同步规则：

- 请求携带属于当前单据的已有 `id`：更新。
- 请求不带 `id`，或 `id` 不属于当前单据：新增。
- 数据库已有、但本次请求未带回的子记录：物理删除。
- 删除行程时，同时删除该行程的每日补助。

请求示例：

```http
PUT /api/reimbursements/CLBX202606100001
Content-Type: application/json
```

请求 Body 可使用 1.4.5 的完整结构；更新已有子项时补充对应 `id`。

### 6.2 响应示例

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "reimNo": "CLBX202606100001",
    "billStatus": "DRAFT",
    "billStatusName": "草稿"
  }
}
```

### 6.3 当前状态限制与错误

- 已作废单据：返回 `已作废单据不能修改`。
- 当前实现没有限制已提交单据，因此 `SUBMITTED` 单据也能被此接口更新，且不会重新执行完整提交校验。这是现存风险，不是推荐业务规则。
- 其他错误包括单据不存在、城市标准不存在、补助金额越界及数据库异常。

## 15. 当前接口限制

- 当前没有登录认证、用户身份或数据权限控制。
- 请求 DTO 未使用 Bean Validation 注解，错误主要由 Service 手工抛出。
- 更新接口只禁止修改已作废单据，没有禁止修改已提交单据。
- 草稿提交接口没有检查原状态必须为草稿。
- 作废接口没有禁止重复作废。
- 分摊类型后端仍兼容 `DEPARTMENT`，尚未强制只能为 `COMPANY`。
- 当前不校验同一人员的多条行程是否发生日期重叠，该问题暂由后续审核流程处理。
- 分页接口没有限制 `pageSize` 最大值。
- Kafka 只有生产端发送逻辑，项目中没有对应消费者。
- Redis 目前主要用于单号序列，没有用于城市标准缓存或接口幂等。
